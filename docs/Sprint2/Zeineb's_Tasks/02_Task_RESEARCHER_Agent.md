# Description

The Researcher Agent is the second node in the LangGraph pipeline. Its job is to take the sub-queries generated by the Planner and fetch candidate sources from both arXiv and the web, running everything concurrently rather than sequentially.

## Key Responsibilities

- **Parallel Search** — Execute arXiv and web searches for each sub-query simultaneously.
- **Source Fetching** — Combine academic papers (arXiv) and web content (DuckDuckGo + BeautifulSoup).
- **Graceful Degradation** — If one search fails, others continue. Never crash the pipeline.
- **Deduplication** — Remove duplicate sources by URL.
- **Source Capping** — Limit total sources to **8**, matching the project's success metric.

---

## Why This Matters

Without the Researcher Agent, the system would only have stub sources. This is the first agent that retrieves real external data, making the pipeline fully functional.

## Core Logic

### `researcher_node()`

The main LangGraph node function.

It:

- Takes `sub_queries` from the graph state.
- For each sub-query, launches **two parallel tasks**:
  - arXiv search (via `ArxivClient`)
  - Web search (via `web_search` + `WebScraper`)
- Collects all results.
- Deduplicates sources by URL.
- Caps the total number of sources at **8**.
- Returns the sources and updates the graph state.

---

## Async Handling

The arXiv client is synchronous (using **tenacity** and internal rate limiting). To avoid blocking the event loop, searches are executed in worker threads.

```python
await asyncio.to_thread(arxiv_client.search, query, max_results)
```

This allows arXiv and web searches to execute concurrently instead of sequentially.

---

## Web Search Flow

1. **Search** — Retrieve search hits (title, URL, snippet) from DuckDuckGo.
2. **Scrape** — Fetch and clean the page content using `WebScraper`.
3. **Filter** — Discard pages that fail to scrape (paywall, timeout, non-HTML).

---

## Error Handling

- If a sub-query returns no results, it is skipped (not fatal).
- If a scraper fails, the corresponding URL is dropped (not fatal).
- If all searches fail, the graph state's `error` field is populated instead of raising an exception.

---

## Deduplication

The same URL may appear across multiple sub-queries. The Researcher Agent keeps only the first occurrence.

---

## Source Capping

The agent limits the total number of sources to:

```python
MAX_TOTAL_SOURCES = 8
```

This matches the project's success metric of **3–8 sources per query**.

# Integration with Existing Clients

The Researcher Agent integrates three existing components.

## 1. ArxivClient

**File**

`src/clients/arxiv_client.py`

**Method**

```python
search(query, max_results) -> List[SourceSchema]
```

**Characteristics**

- Synchronous
- Uses **tenacity** for retries
- Throttled to approximately **1 request every 3 seconds** (arXiv policy)

---

## 2. `web_search`

**File**

`src/clients/brave_client.py`

**Function**

```python
web_search(query, count) -> List[WebSearchResult]
```

**Characteristics**

- Asynchronous
- Uses DuckDuckGo (free, no API key)
- Automatic fallback if Brave search fails

---

## 3. WebScraper

**File**

`src/clients/scraper.py`

**Method**

```python
fetch(url) -> Optional[SourceSchema]
```

**Characteristics**

- Synchronous
- Uses BeautifulSoup to extract clean page content
- Timeout: **15 seconds per URL**

# Performance Considerations

## Dominant Cost

`ArxivClient` uses `min_interval = 3.0`, meaning each arXiv request is throttled to roughly one request every three seconds.

With multiple sub-queries, this throttling—not network latency—is expected to be the dominant cost of the Researcher Agent.

---

## Parallelization Benefit

Using `asyncio.gather()` together with `asyncio.to_thread()` allows all searches to execute concurrently.

For **3 sub-queries**:

| Execution Mode | Approximate Runtime |
|----------------|--------------------|
| Sequential | ~9 seconds (3 × 3 seconds) |
| Parallel | ~3 seconds |
